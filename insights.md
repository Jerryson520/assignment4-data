# Insights

记录做这份作业过程中值得沉淀的通用认知与工程模式。

## 1. 把"分类器一定会犯错"当作系统设计的前提

来自 language_identification (b)：数据过滤流水线里的任何分类器（语言识别、质量分类、毒性过滤）都不可能 100% 准确，可靠的系统不是假设分类器是对的，而是围绕"它会犯错"来设计三道防线：

**上线前：离线评估要贴近真实分布。**
官方汇报的准确率（如 fastText 97%+）是在干净长文本上测的；真实网页里最容易翻车的是短文本、混合语言、相近语言（马来语 vs 印尼语）、非语言内容（代码/表格）。验证集必须按真实数据的分布构建，甚至超采样这些难例，否则"99% 准确率"上线即翻车。

**决策时：分层处理，而非一刀切。**
阈值不是拍脑袋定的，要在验证集上画阈值—精确率/召回率曲线来选。高阈值宁缺毋滥但会系统性误杀某些人群的语言变体（方言、口语、小语种），造成偏差。更好的做法是分层：高置信直接收；低置信交给第二个独立识别器复核（两个独立检测器同时犯同一个错的概率远低于单个——同医学上的二次检查）；仍拿不准的人工抽检。

**运行中：监控分布 + 端到端验证兜底。**
单条判错不可怕，可怕的是系统性偏移悄悄积累。每批数据统计过滤前后的语言分布和丢弃率，异常偏移（英文占比骤降、丢弃率飙升）意味着上游出事（识别器版本变化、新型垃圾页爆发、解码 bug）。最后不信任任何中间指标：用过滤后的数据训一个小规模代理模型，在多语言基准上与上一版对比——即使所有中间环节"看起来正常"，最终能力下降也能被抓住。

一句话：**离线评估（贴近真实分布）→ 分层决策（复核而非丢弃）→ 运行监控 + 端到端验证**，这个三段式适用于流水线里的每一个分类器。

## 2. 置信度阈值只在"错误与低置信相关"时才有用

来自 language_identification (c) 与 harmful_content (d) 的对比实验，同一批 20 条 WARC 样本：

- **语言识别**：唯一的错误（乱码韩语页判成 en）置信度 0.27，而所有正确预测在 0.46–0.99——错误与低置信强相关，画一条 0.5 的阈值线就能把错误滤掉。阈值有效。
- **有害内容分类**：Jigsaw NSFW/毒性分类器漏报了全部 3 个成人页面，且漏报时置信度高达 0.978–0.999（non-nsfw），与正常页面完全混在一起。阈值无效。

差别的根源是**分布匹配度**：fastText lid.176 的训练数据覆盖了大多数真实网页语言，它"知道自己什么时候不确定"；Jigsaw 分类器在英文维基百科评论（短用户留言）上训练，面对中文页面和"导航骨架+零散露骨词"的网页文本是彻底的分布外（OOD）输入——**分布外的模型不是不确定，而是自信地胡说**（连正文含 "sexually explicit material" 的英文成人站都判 non-nsfw 0.999）。

**Why:** 置信度是模型在自己训练分布内校准出来的量；离开分布，校准失效，置信度不再携带"对错"信号。

**How to apply:** 拿到一个分类器，先问它的训练分布和你的数据分布差多远，再决定阈值策略有没有意义。错误集中在低置信区 → 调阈值；错误高置信出现 → 阈值救不了，该换/重训分类器（多语言模型、在目标域数据上微调），或按语言/域先路由再分类。判断方法就是 insight 1 的那一步：人工标注小样本，把"置信度 vs 对错"排序画出来看。
