# Insights

记录做这份作业过程中值得沉淀的通用认知与工程模式。

## 1. 把"分类器一定会犯错"当作系统设计的前提

来自 language_identification (b)：数据过滤流水线里的任何分类器（语言识别、质量分类、毒性过滤）都不可能 100% 准确，可靠的系统不是假设分类器是对的，而是围绕"它会犯错"来设计三道防线：

**上线前：离线评估要贴近真实分布。**
官方汇报的准确率（如 fastText 97%+）是在干净长文本上测的；真实网页里最容易翻车的是短文本、混合语言、相近语言（马来语 vs 印尼语）、非语言内容（代码/表格）。验证集必须按真实数据的分布构建，甚至超采样这些难例，否则"99% 准确率"上线即翻车。

**决策时：分层处理，而非一刀切。**
阈值不是拍脑袋定的，要在验证集上画阈值—精确率/召回率曲线来选。高阈值宁缺毋滥但会系统性误杀某些人群的语言变体（方言、口语、小语种），造成偏差。更好的做法是分层：高置信直接收；低置信交给第二个独立识别器复核（两个独立检测器同时犯同一个错的概率远低于单个——同医学上的二次检查）；仍拿不准的人工抽检。

**运行中：监控分布 + 端到端验证兜底。**
单条判错不可怕，可怕的是系统性偏移悄悄积累。每批数据统计过滤前后的语言分布和丢弃率，异常偏移（英文占比骤降、丢弃率飙升）意味着上游出事（识别器版本变化、新型垃圾页爆发、解码 bug）。最后不信任任何中间指标：用过滤后的数据训一个小规模代理模型，在多语言基准上与上一版对比——即使所有中间环节"看起来正常"，最终能力下降也能被抓住。

一句话：**离线评估（贴近真实分布）→ 分层决策（复核而非丢弃）→ 运行监控 + 端到端验证**，这个三段式适用于流水线里的每一个分类器。
