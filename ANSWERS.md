## Problem (look_at_cc)

### (a)

第一个网页的 URL 是 `http://0371rykj.com/ipfhsb/34.html`。该页面现在已无法访问：原 URL 返回 HTTP 301 跳转到 `www.0371rykj.com`，而该域名已无法解析，页面实际失效。从原始 HTML 中可见的中文文本（"恒温恒湿试验箱"、"上海林频仪器股份有限公司"及"试验箱系列 / 老化箱系列 / IP防护系列"等导航菜单）可以看出，这是一家仪器公司的中文产品介绍页面。

### (b)

- **哪些部分应该被提取器过滤掉？** 开头两行是 `<title>`/`<h1>` 中解码出来的成人垃圾 SEO 关键词；其后的导航菜单（首页/林频产品/试验箱系列/新闻中心…）、面包屑路径（"您所在的位置：…"）、服务热线和手机号、"了解详情 立即咨询"等按钮文字——这些都是页面骨架而非正文，都应被过滤。
- **用这种文本训练会出什么问题？** 模型会学到在无数页面中重复出现的导航样板短语，浪费容量并助长复读式输出；会把成人垃圾关键词与正常产品文本错误关联；还可能记住电话号码等联系信息，带来隐私风险。
- **模型能从中学到什么有用信息？** 页面中间的正文（"恒温恒湿试验箱是航空、汽车、家电、科研等领域必备的测试设备…"）和型号参数表（LRHS-101-LH 450×450×500 等）是真实且有信息量的，模型可以学到环境试验设备领域的中文术语、产品描述方式和规格表达。

### (c)

对于面向中文工业/制造业场景的应用（如工业设备电商搜索、B2B 采购客服、仪器领域问答），这个例子是有用的训练数据，因为它包含真实的设备术语、型号和规格参数。而对于面向普通消费者的通用对话助手或内容安全要求高的应用（如儿童教育产品），它就不合适——正文之外混入的成人垃圾关键词和大量导航样板文字会带来有害关联和低质量的语言模式。

### (b)

在同一个 WARC 分片的前 20 条记录上运行我的提取函数，并与对应 WET 记录逐条对比（见 results/my_extract.txt 与 results/wet_extract.txt）。

**相似之处：**
- 两者提取出的正文内容基本一致——导航菜单、正文段落、规格参数表都同时出现在两边；
- 都没有过滤导航、电话等样板内容，都需要后续的质量过滤。

**差异：**
- WET 包含 `<title>` 的内容（如第 3 条开头的 "Welcome to USNCCM13!"），resiliparse 不含 title；
- WET 每个文本块紧凑地占一行；resiliparse 保留列表层级结构（"•" 项目符号和缩进），但夹杂大量空行；
- resiliparse 会把隐藏垃圾块里的原始 HTML 标签（如 `<th id="gckmo">`）当作文本泄漏出来。

**哪个更好：** WET 的输出更干净紧凑；resiliparse 保留的结构信息更多，且可通过 main_content=True 等选项进一步做主内容提取，可控性更强。

## Problem (language_identification)

### (a)

见 tests/adapters.py 的 `run_identify_language` 函数（通过 `uv run pytest -k test_identify_language`）。

### (b)

**可能出现的问题：**
- 误判会把目标语言的高质量文本错误丢弃——短文本、方言、混合语言、含大量代码/专有名词的页面上识别器最不可靠；
- 其他语言或乱码被混进语料；
- 下游后果：模型在低资源语言上表现更差、输出夹杂外语、学到乱码模式。

**高风险场景（面向用户的产品）的缓解措施：**
- 在人工标注的多语言验证集上评估识别器，重点测短文本、混合语言等边界情况；
- 根据下游需求校准置信度阈值；对低置信样本用第二个独立识别器复核或人工抽检，而非一刀切丢弃；
- 持续监控过滤前后的语言分布和丢弃率，异常偏移时回溯检查；
- 用过滤后的数据训练模型并做多语言下游评测，端到端验证整个过滤流程。

## Problem (mask_pii)

### (a) (b) (c)

### (d)

**朴素套用会带来三类下游问题：**
- **误杀（false positive）**：版本号、订单号、日期等长得像电话/IP 的字符串被替换掉，破坏正常文本（如 "Python 3.10.2.146" 可能被吃掉一半）；
- **漏网（false negative）**：正则只覆盖美式电话等有限格式，世界各地的写法（+86 138xxxx、带国家码或空格分组）大量漏掉，隐私保护目标并未真正达成；
- **占位符污染**：`|||PHONE_NUMBER|||` 这类人造高频 token 混入训练分布，模型可能在推理时原样吐出占位符，或学会"此处该有个电话号码"而幻觉编造出真实号码。

**缓解方法：**
- 用更可靠的检测器替代纯正则：libphonenumber 做电话解析验证、NER 模型识别人名地址等更广义的 PII；
- 在人工核查的验证样本上量化误杀率和漏网率，再据此调整规则；
- 替换时不用显眼占位符，改用格式合法的虚构值（如保留区号结构的假号码）；
- 对 PII 密度过高的文档整篇丢弃，而非逐个替换。
